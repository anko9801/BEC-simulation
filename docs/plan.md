## システムの二層構造

このソフトウェアが本質的に**二つの物理レイヤー**を持っていることが最初の重要な認識です。この二層構造が全ての設計判断の出発点です。重要なのは**光学系層とBEC層はほぼ独立して開発できる**が、接続部分（`LaserBeamPotential`, `RamanCoupling`）だけインターフェースを最初に合意する必要があるということです。

---

## 実装順序の設計思想

実装順序を決める基準は4つあります。

**基準A: 依存関係**（Bを実装するにはAが必要）
**基準B: 即時実験価値**（今すぐあんこさんの実験に役立つか）
**基準C: リスク除去**（後で作り直しを強いられるか）
**基準D: 並列可能性**（同時に進められるか）

この4基準で評価した結果の実装ロードマップです。---

## 各ステージの設計思想と技術的決断

### Stage 0 — Foundation（なぜ最初か）

ここを先に決めないと後で全部作り直しになる部分です。

**最重要決断：単位系の戦略**

SpinorBEC.jlは現在`DimensionlessScales`という自前の単位変換を使っています。これは計算速度的に正しい選択です。問題は「何を入力するか」の境界です。提案は：

```julia
# 入力はUnitful.jlで自然に書ける（460u"nm"）
# 内部計算は全部無次元（実行時ゼロオーバーヘッド）
# 境界で一度だけstrip_units()する

laser = LaserSource(
    wavelength = 460u"nm",   # ユーザーが書きやすい
    power      = 3.5u"mW",
    waist      = 0.75u"mm"
)
# 内部では460e-9, 3.5e-3, 7.5e-4として計算
```

**テスト戦略**

各モジュールに必ず「解析解と比較するテスト」を入れる。例えばABCD伝播なら「ビームウェストが焦点面に来るか」という自明な計算で検証する。これがないと後でバグを発見した時に何が壊れているか分からなくなります。

### Stage 1 — AtomOptics.jl コア（なぜ最初に作るか）

**理由1: 今すぐ実験で使える**

ファイバーカップリングの最適化で毎回手計算していたABCD行列の計算が自動化されます。ミラー位置のスキャン結果（+2mm, +4mm...）をシミュレーションと比較できるようになります。

**理由2: SpinorBEC.jlの`CrossedDipoleTrap`が改善される**

現在のSpinorBEC.jlの`GaussianBeam`は Rayleigh長を無視した簡易モデルです。AtomOptics.jlの`GaussianBeam`で置き換えると、より現実的なトラップポテンシャルになります。

**最も重要な設計：`1a. GaussianBeam`**

複素ビームパラメータ q を使う実装が本質的に正しいです。

```julia
struct GaussianBeam
    q::ComplexF64      # 複素ビームパラメータ
    λ::Float64         # 波長 [m]
    P::Float64         # パワー [W]
    M2::Float64        # ビーム品質因子（理想=1）
end

# ABCDによる伝播は行列積一つ
function propagate(beam::GaussianBeam, M::SMatrix{2,2,Float64})
    q_new = (M[1,1]*beam.q + M[1,2]) / (M[2,1]*beam.q + M[2,2])
    return GaussianBeam(q_new, beam.λ, beam.P, beam.M2)
end

# ウェスト半径（これが実験で測るビーム径）
waist_radius(b::GaussianBeam) = sqrt(-b.λ / (π * imag(1/b.q)))
```

M²因子を入れることが重要で、実際の460nmビームはM²=1.1程度のはずです。これを無視すると計算と実験が5-10%ずれます。

**`1c. FiberCoupler`の核心**

モード重複積分はガウシアン同士なら解析解があります：

```
η = |⟨E_fiber | E_beam⟩|² / (‖E_fiber‖² ‖E_beam‖²)
```

ガウシアン同士の場合はビームウェストの比と位置ずれの関数として閉じた形で書けます。これが実装できると「今のカップリングが44%な理由」を定量的に説明できるようになります。

### Stage 2 — SpinorBEC.jl 補強（Stage 1と並列）

現在のコードで**一番深刻な問題**から優先順位を付けます。

**最優先: `2b. DDI修正`**

アーキテクチャドキュメントを見ると、DDIカーネル `Q_ab(k) = k_a*k_b/k² - δ_ab/3` は k=0 で 0/0 不定形になります。現状どう処理しているか不明ですが、k=0成分をゼロに設定する正規化が必要です（これは双極子相互作用の物理的要請でもあります）。もし間違っていたら虚数時間伝播が発散します。

**高価値: `2c. Thomas-Fermi初期状態`**

現在は単純なガウシアンで初期化してから虚数時間で収束させています。トラップ密度が高い極限ではThomas-Fermi（運動エネルギー無視）の解析解が使えます：

```
n_TF(r) = max(0, (μ - V(r)) / g)
```

これを初期状態にすると虚数時間の収束に必要なステップ数が10倍以上減ります。

**構造的改善: `2d. 適応的dt`**

虚数時間伝播の収束チェックは現在「save_everyステップごとにエネルギーを比較」しています。より良いのは**ライン探索**で各ステップのdtを最適化することです。計算コスト増加を抑えながら収束を速くできます。

### Stage 3 — 接続層（最も設計が難しい部分）

ここが「光学系シミュレーション」と「BECシミュレーション」をつなぐ心臓部です。

**`LaserBeamPotential`の設計**

```julia

end

    # V(r) = -α I(r) / (2ε₀c)
    # ← ここでAtomOptics.jlのGaussianBeamを使う
end
```

これで現在の`CrossedDipoleTrap`の内部実装が`AtomOptics.jl`を使う形に自然に置き換わります。

**`RamanCoupling`（最も物理的に重要）**

スピノルBECの実験でゼーマン効果と並んで重要なのがラマン結合です。二本のレーザービームが異なるスピン状態間をコヒーレントに結合します：

```julia
struct RamanCoupling
    Ω_R::Float64    # ラビ周波数
    δ::Float64      # 二光子デチューニング
    k_eff::SVector  # 有効波数ベクトル（ビームの差）
end
```

これはゼーマン効果と同じく対角要素への追加としてsplit-step伝播に組み込めます。

---

### Stage 4 — GPU対応（後回しで良い理由）

GPU対応を後にする理由は、**バグの発見がCPUの方が圧倒的に楽**だからです。正しい物理が出るまでCPUで開発し、速度が必要になったらKernelAbstractions.jlでGPUバックエンドに切り替えます。

`KernelAbstractions.jl`の美しさは、同じカーネルコードがCPU/CUDA/Metal/ROCmで動くことです。FFTは`AbstractFFTs.jl`のインターフェース経由で書けば、`FFTW.jl`（CPU）と`CUDA.jl`のCuFFTが自動的に切り替わります。

## 実装順の中で特に「最初に決めること」---

## SpinorBEC.jlの現状コードで直すべき具体的な問題点

アーキテクチャドキュメントを精査した結果、以下の問題が見つかりました。

**問題1: `spin_mixing.jl`の性能**

現状は「各格子点でHermitian行列を固有値分解してmatrix expを計算」しています。スピン1（3×3行列）では対称化された行列のトレースから解析的固有値が求まります（Cardanoの公式）。これを使うと固有値分解が不要になり2-3倍速くなります。スピン6（Eu151, 13×13）では固有値分解が必須ですがキャッシュ戦略が重要です。

**問題2: `_component_slice`によるビュー生成のオーバーヘッド**

内部ループで毎回`view(psi, idx...)`を作るとJuliaのGCに負荷がかかります。ワークスペース初期化時にビューを全部事前に作って格納しておく方が速いです。

**問題3: DDI k=0の処理（前述）**

**問題4: 実験YAML の`potential`指定の柔軟性不足**

現在の仕様だと`CrossedDipoleTrap`でビームパラメータをYAMLで指定できるようになっていますが、波長・パワー・ウェストがAtomOptics.jlのGaussianBeamと同じパラメータセットになっていません。AtomOptics.jl完成後に統一する必要があります。

## 「完璧」の定義とその検証方法

最後に、「完璧なレベル」を何で測るかを決めておくことが重要です。

| 検証項目           | 手法                                                               |
| ------------------ | ------------------------------------------------------------------ |
| BEC物理の正しさ    | Thomas-Fermiプロファイルと解析解比較、ソリトンの数値解と既知解比較 |
| 光学系の正しさ     | 実験のカップリング効率（48%）をパラメータで再現できるか            |
| AOM効率の正しさ    | 3080-125@460nm の実測値（63%）と予測値の一致                       |
| 偏光モデルの正しさ | 消光比374:1の再現、最適角267°の再現                                |
| 性能               | 2D 256×256 スピン1 が1ステップ < 10ms（CPU）、< 1ms（GPU）         |
